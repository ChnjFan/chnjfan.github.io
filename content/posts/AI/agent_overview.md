## 什么是 Agent？

Agent 是指 LLM 在循环中自主调用工具，直到完成用户指定的任务。

本质上 Agent 就是一个能自主完成目标的 AI 系统，跟传统 LLM 的核心区别就在于「自主性」和「能行动」上。传统 LLM 用户问一个问题回答一个问题，每次都是独立被动响应的；Agent 有自己的规划能力，对于一个复杂目标，会将任务拆分成多步，通过调用工具、访问记忆、感知环境来一步一步执行直到完成。

普通大模型的局限性：
1. 普通大模型无真实感知和主观意识，本质是一个问答机器，就算是多轮会话也只是在当前上下文被动响应，不会主动做事。
2. 幻觉问题，模型训练数据有截止日期，没有途径取获取最新的信息，只能靠概率生成看似专业实则错误的答案。
3. 不能自主行动，只是文本输出，没办法根据文本去执行具体的操作，如果你让大模型帮你发送邮件或查询数据库，它只会告诉你怎么做。

Agent 特别之处在于它有一个核心的运作闭环：感知 -> 规划 -> 行动 -> 再感知，可以让 AI 拥有拆解任务，执行任务和反馈结果的能力。

1. **工具调用**：这是让 Agent 从「说话」变成「做事」的关键。Agent 能调用外部工具，比如搜索引擎、代码执行器、数据库、API 等等。模型告诉 Agent 应该调什么工具，Agent 执行后将结果反馈给模型。
2. **记忆机制**：传统大模型每次对话都是失忆的，除非手动传上下文。Agent 系统通常会设计**短期记忆**和**长期记忆**两层。短期记忆就是当前任务执行过程中的中间状态，比如第一步搜索到了什么、第二步计算结果是多少，这些都存在上下文里，保证 Agent 不会做到一半忘了前面发生了什么。长期记忆则是跨任务的，比如用户的偏好、历史操作记录，通常用向量数据库来存储，需要的时候做语义检索拿回来。有了这两层记忆，Agent 在执行复杂任务时才能保持连贯性，不会走着走着忘了目标是什么。
3. **多步推理和自我纠错**：区别于自动化脚本，感知失败后通过分析原因，换一种方式重试。


{{< notice note >}}
Agent = LLM + Harness
Harness 是执行指定任务时，在正确的时机为模型提供合适的上下文信息；是围绕 LLM 在循环过程中的中间件。
{{< /notice >}}


## Agent 的核心组件

Agent 的基本架构有四个核心组件：**LLM**、**工具**、**记忆**、**规划**模块。
- LLM 是整个系统的大脑，负责理解任务和做决策；
- 工具让 Agent 能跟外部世界交互，搜索、执行代码、调 API 都靠它；
- 记忆让 Agent 在任务执行过程中保持状态，不会「失忆」；
- 规划模块负责把复杂目标拆解成可执行的步骤。


### LLM

LLM 是 Agent 系统中的推理引擎，所有的输入（用户输入、工具结果、记忆内容）都要经过 LLM 来理解、决策并提供最终答案。

除了文本生成外，大部分 LLM 支持：
- 工具调用：调用外部工具并在回复结果。
- 结构化输出：模型输出限制为指定格式。
- 多模态：除了文本之外还支持视频、音频和图片等数据。
- 推理：模型通过多步推理得到结论。

模型的质量和性能直接决定了 Agent 的基础可靠性和表现。

不同模型擅长的任务各不相同——有些更擅长遵循复杂指令，有些更擅长结构化推理，还有些支持更大的上下文窗口，以处理更多信息。

**System Prompt**（系统提示词）在 Agent 开始工作前设定好模型的行为方式。你可以使用系统消息来设定语气、定义模型的角色，并为回复内容格式制定准则。


### 工具系统

工具系统是 Agent 与外界交互的唯一方式，能够让大模型获取实时数据、执行代码、查询外部数据库，让大模型拥有自主行动的能力。

在底层，工具是具有明确定义的输入和输出的可调用函数，这些输入和输出会传递给大模型，模型根据上下文决定何时调用工具，以及提供哪些输入参数。

用户与模型之间的基本工具调用流程：

![20260708155331113.svg](https://raw.githubusercontent.com/ChnjFan/img-bed/master/img/20260708155331113.svg)

使用 `@tool` 装饰器创建工具，函数的文档会作为工具的描述，帮助大模型理解何时使用该工具。

```python
from langchain.tools import tool

@tool
def search_database(query: str, limit: int = 10) -> str:
    """Search the customer database for records matching the query.

    Args:
        query: Search terms to look for
        limit: Maximum number of results to return
    """
    return f"Found {limit} results for '{query}'"
```

OpenAI function calling 的工具描述格式如下：

```json
# 你只需要告诉模型三件事：工具叫什么名、能做什么事、需要哪些参数
tools = [
    {
        "type": "function",
        "function": {
            "name": "search_database",
            "description": "在客户销售数据库中检索与查询条件匹配的记录",
            "parameters": {
                "type": "object",
                "properties": {
                    "query": {
                        "type": "string",
                        "description": "搜索关键词"
                    },
                    "limit": {
                        "type": "int",
                        "description": "返回的最大条目数"
                    }
                },
                "required": ["query"]
            }
        }
    }
]

# LLM 决定调用工具时，会返回类似这样的结构：
{"tool_call": {"name": "search_database", "arguments": {"query": "2024年大模型最新进展", "limit": 10}}}
# 然后你的代码负责真正执行这个搜索，把结果再塞回给 LLM
```

工具描述的质量直接决定了 Agent 的表现，模型根据工具描述判断什么时候调用工具，如果描述写的模糊有歧义，模型就可能在错误的时机调用，或者不调用工具。

例如工具库描述中写了查询数据，那么用户在询问天气数据时也会去调用工具查数据库，因为描述太过宽泛。如果像上面的示例中定位在客户数据库中检索记录，那么模型就能判断出销售数据查询需要调用该工具。

MCP（Model Context Protocol，模型上下文协议）基于 JSON_RPC 通信协议定义了工具调用的标准，工具提供方只需要按照 MCP 标准实现一个 Server，任何支持 MCP 的 Agent 都可以直接根据 MCP 标准调用工具。

MCP 架构分为三层：
1. Host：用户交互的 AI 应用。
2. Client：管理 AI 应用与 MCP Server 的连接。
3. Server：工具提供商提供给的工具能力。


### 记忆系统

记忆系统用来存储过往交互的信息，主要分为**短期记忆**和**长期记忆**。

#### 短期记忆

短期记忆主要记录当前这轮对话中的上下文，装载 context window 中。

Agent 在一次任务执行过程中靠它记住中间状态，比如第一步搜索到了什么、第二步执行结果是什么。这就像人的「工作记忆」，容量有限，任务一结束就清空了。

#### 长期记忆

长期记忆通过向量数据库实现，将重要信息编码后保存，用的时候做语义检索拿出来。Agent 需要主动查询才能调用。

短期记忆的问题就是上下文窗口的限制，一个复杂任务执行十几步，每步工具返回大量文本，上下文很快就满了，这时候你要么做摘要压缩（把前面的步骤浓缩成关键信息），要么做滑动窗口（只保留最近几步的详细内容），但不管哪种方式都会丢失信息。

长期记忆的挑战则在于「什么该存、什么不该存」以及「存了之后怎么管理」。如果什么都往向量数据库里塞，检索出来的噪音会很多，反而干扰模型的决策；如果存得太少，又失去了长期记忆的意义。存入之前做一轮重要性评估，只把真正有价值的信息持久化。

记忆衰减（Memory Decay）：给每条记忆增加时间权重，越久远的记忆权重越低，检索的时候自然就排在后面了。

```
记忆的相关性分数 = 语义相似度 x 时间衰减因子
```

### 规划模块

规划模块决定了 Agent 处理复杂任务的能力。

规划模块的底层还是依赖大模型的推理能力，提升推理能力的方式有：
1. 思维链（Chain of Thought）：让模型把思考过程写下来。LLM 的 token 生成是逐步进行的，每一步推理的输出会成为下一步推理的输入，把中间步骤写出来，等于给了模型更多的「思考空间」。
2. 思维树（Tree of Thought）：每个推理节点上展开多个可能的分支，然后评估每个分支的质量，选出最优的路径继续往下走。

模块架构主要两种模式：

| 模式 | 简述 | 优点 | 缺点 |
| --- | --- | --- | --- |
| **Plan-and-Execute** | 先让 LLM 输出一个完整的步骤列表，然后按照顺序逐步执行。  | 整体结构清晰，执行前就能看到计划   | 中间某一步执行结果不达预期后，需要重新调整原来计划 |
| **ReAct**            | 每走一步就根据当前结果重新思考下一步该做什么，不提前制定完整计划。 | 灵活性极高，能根据实际情况随时调整 | 容易「走偏」，因为每一步都是局部最优决策，有时候会忽略整体目标 |


各模块之间整理运行的逻辑用伪代码表示：

```python
# Agent 运行的核心 loop（伪代码）
def agent_run(user_goal: str):
    # 第一步：规划模块上场，把目标拆成步骤列表
    plan = llm.plan(user_goal)

    memory = []  # 短期记忆，用来存每一步的中间结果

    for step in plan:
        # 第二步：LLM 核心做决策，这一步该怎么做？
        action = llm.decide(
            step=step,
            history=memory,                    # 把短期记忆传进去，让它知道之前做了什么
            long_term=vector_db.search(step)   # 从长期记忆里捞出相关历史
        )

        if action.type == "tool_call":
            # 第三步：工具系统负责真正执行
            result = tools.execute(action.tool_name, action.args)
            memory.append({"step": step, "result": result})  # 执行结果存入短期记忆

        elif action.type == "final_answer":
            return action.content  # LLM 判断任务完成，返回最终答案
```
